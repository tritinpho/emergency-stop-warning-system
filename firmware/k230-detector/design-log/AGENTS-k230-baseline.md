# K230 firmware baseline

- Treat `k230-firmware/` as the authoritative, device-tested runtime baseline for all generated K230 code in this repository. Before generating or changing K230 code, inspect the corresponding implementation in `k230-firmware/libs/`, `k230-firmware/apps/`, or `k230-firmware/src/` and preserve its API signatures, argument order, resource lifecycle, and hardware-specific defaults unless the task explicitly requires a deviation.
- Target the Yahboom SD-card image identified by `k230-firmware/Version.txt`: image release `1.4.1`, released `2025-08-20`, with nncase/kmodel toolchain `2.9.0`. Do not infer the image version from `configs/sys_config.json`: `SYS_INFO.fw_version` is GUI metadata and is not the SD-card image release. The same config identifies GUI OS `YAHBOOM K230 GUI 2.0` and hardware revision `1.4`.
- Use the bundled `libs` implementation as the compatibility contract. In particular, K230 vision code uses `PipeLine`/`AIBase`/`Ai2d`, camera channel 2 in planar RGB888 for inference, 16-pixel width alignment, ST7701 LCD mode at `640x480` by default, and explicit `deinit()`/`destroy()` cleanup.
- Keep width/height conventions exact: application sizes are `[width, height]`, AI2D tensor shapes are `[1, 3, height, width]`, and `aidemo` postprocess size arguments are `[height, width]`. Use `letterbox_pad_param` plus padding value `[128, 128, 128]` for YOLO detection preprocessing, then `tf_bilinear` with `half_pixel`, matching the bundled YOLO examples.
- Keep model, preprocessing, labels, and postprocessing parameters as one verified set. For bundled COCO YOLOv8 examples, use the matching `.kmodel` input size, all 80 labels, `aidemo.yolov8_det_postprocess`, and a color table generated for every label. Never mix a model from another firmware/toolchain version without reconverting and validating it on the device.
- Validate generated code against both the relevant file in `k230-firmware/` and the matching chapter in `K230-docs/`. When they differ, prefer the executable firmware baseline for runtime behavior, document the discrepancy, and do not silently modernize APIs from upstream CanMV examples.
